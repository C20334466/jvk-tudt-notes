Keyboard Accessibility

Introduction: Keyboard accessibility ensures that users can navigate and interact with the webpage using only a keyboard. This is especially important for users who rely on assistive technologies or prefer not to use a mouse. The following updates were made to the page to improve keyboard accessibility for interactive elements like links, buttons, and headers.

Description of Code Changes

The focus was on adding appropriate `tabindex` values and keyboard event handlers for actionable elements. Below is a comparison of the original and updated code:

    <p>Asynchronous programming with callbacks</p>
    <p>Asynchronous programming with promises</p>
    <p>Multithreading</p>

New Code:

    <p tabindex="0">Synchronous programming</p>
    <p tabindex="0">Asynchronous programming with callbacks</p>
    <p tabindex="0">Asynchronous programming with promises</p>
    <p tabindex="0">Multithreading</p>

    <script>type="text/javascript" src="../common/res/scripts/main.js"</script>

    // Function to toggle the visibility of the article
	function toggle_article(article) {
		if (article) {
			if (article.style.display === 'none') {
				article.style.display = 'block'; // Open article
			} else {
				article.style.display = 'none'; // Close article
			}
		}
	}

	// Function to handle keyboard events (Enter, Escape)
	function handle_keyboard_navigation(e) {
		const header = e.target;
		const article = header.nextElementSibling; // Get the next sibling (article)

		// Handle the Enter key - open/close the corresponding article
		if (e.key === "Enter") {
			toggle_article(article);
		}

		// Handle the Escape key - close the article if it's open
		if (e.key === "Escape") {
			if (article && article.style.display !== 'none') {
				toggle_article(article);
			}
		}
	}

	// Initialize event listeners for keyboard navigation
	function init_keyboard_navigation() {
		const headers = document.querySelectorAll('p[tabindex="0"]');
		headers.forEach(header => {
			header.addEventListener('keydown', handle_keyboard_navigation);
		});
	}

	window.addEventListener("load", () => {
		init_keyboard_navigation(); // Initialize keyboard navigation after the page loads
	});

How the Solution Was Reached

The first step was to identify interactive elements in the header that required keyboard accessibility. Elements like the title and navigation links were given a `tabindex="0"` attribute, enabling them to be focused using the Tab key. I consulted with ChatGPT to refine this approach and ensure comprehensive coverage of accessibility best practices.

Event listeners were implemented for `keydown` events to handle actions triggered by pressing the Enter key. This allows users to interact with elements in a meaningful way. The implementation of these event listeners was guided by suggestions from ChatGPT, which helped optimize the handling of keyboard interactions.

What Was Easy and What Was Hard

The easy part was adding `tabindex` and ARIA roles, as these changes are straightforward and well-documented. The challenge was handling keyboard events consistently across browsers and ensuring that no unintended side effects were introduced (e.g., duplicate actions). Testing with screen readers and browser tools was essential to validate the implementation.

Links to Element IDs

Introduction: Enhancements were made to ensure that links pointing to specific element IDs function correctly, even when the target section is collapsed. This ensures seamless navigation for users.

Description of Code Changes

Code changes focused on ensuring that anchor links expand collapsed sections and smoothly scroll to the target. Below is a comparison of the original and updated implementations:

Old Code:

<a href="#section1">Go to Section 1</a>
<article id="section1">
    <p>Content of Section 1</p>
</article>

New Code:

<a href="#section1">Go to Section 1</a>
<article id="section1" style="display: none;">
    <p>Content of Section 1</p>
</article>

<script>
function toggle_article(article, forceOpen = false) {
    if (article) {
        article.style.display = forceOpen ? 'block' : (article.style.display === 'none' ? 'block' : 'none');
    }
}

function handle_anchor_navigation() {
    document.querySelectorAll('a[href^="#"]').forEach(anchor => {
        anchor.addEventListener('click', function (event) {
            const targetId = this.getAttribute('href').substring(1);
            const targetElement = document.getElementById(targetId);

            if (targetElement) {
                const article = targetElement.closest('article');
                if (article && article.style.display === 'none') {
                    toggle_article(article, true);
                }
                targetElement.scrollIntoView({ behavior: 'smooth' });
            }
        });
    });
}

window.addEventListener("load", handle_anchor_navigation);
</script>

How the Solution Was Reached

The first step was identifying that navigating to a link pointing to a collapsed section failed to display the target content. By introducing JavaScript logic to check the visibility of the target section and force it to open, the problem was resolved.

To improve user experience, smooth scrolling was added to the navigation process. Suggestions from ChatGPT were instrumental in refining this logic, ensuring compatibility across various browsers and use cases.

What Was Easy and What Was Hard

Adding event listeners for anchor links was straightforward. The challenge was implementing logic to handle collapsed sections without introducing unwanted side effects. Additionally, ensuring smooth navigation required careful handling of scrolling behavior, especially for deeply nested elements.
